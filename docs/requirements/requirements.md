# スタジオゼブラ 予約管理システム 要件定義書

## 1. プロジェクト概要

### 1.1 背景
スタジオゼブラは新宿に位置する白ホリゾントを備えた撮影スタジオです。現在の予約プロセスは手動で管理されており、効率化が求められています。

### 1.2 目的
予約管理システムを開発することで、以下の目標を達成します：
- 予約プロセスの半自動化
- 利用者への通知機能の実装
- リピーター向けの利便性向上
- マルチデバイス対応のインターフェース提供
- 広告収入の獲得機会の創出

### 1.3 プロジェクト範囲
本プロジェクトは、MVPフェーズと追加機能フェーズの2段階で実施します。

## 2. ユーザータイプ
- **一般ユーザー**：スタジオを利用する顧客
- **管理者**：スタジオの予約・運営を管理するスタッフ

## 3. 機能要件

### 3.1 MVP機能

#### 3.1.1 ユーザー向け機能
1. **予約カレンダーの確認**
   - 利用可能な日時のオンライン確認

2. **仮予約の申し込み**
   - 予約日の1週間前までの仮予約機能
   - 必要情報の入力と送信

3. **本予約の申し込み**
   - 確定予約の申し込み機能
   - キャンセル料発生の条件表示

4. **予約の変更依頼**
   - 既存予約の日時・内容変更リクエスト

5. **予約の削除依頼**
   - 既存予約のキャンセルリクエスト

#### 3.1.2 管理者向け機能
1. **予約カレンダーの編集**
   - 予約状況の手動更新
   - ブロック時間の設定

2. **仮予約の承認**
   - 申し込まれた仮予約の確認と承認
   - 自動メール通知

3. **本予約の承認**
   - 申し込まれた本予約の確認と承認
   - 自動メール通知

4. **予約変更の承認**
   - 変更リクエストの確認と承認/拒否

5. **予約削除の承認**
   - 削除リクエストの確認と承認/拒否

### 3.2 追加機能（MVP後）

#### 3.2.1 ユーザー向け機能
1. **予約の通知**
   - 仮予約期間終了の事前通知
   - 予約確認のリマインダー

2. **事前決済**
   - オンライン決済システムの統合

3. **キャンセル待ち機能**
   - 満席時のキャンセル待ちリスト登録

4. **利用履歴の記録**
   - 過去の利用状況の確認
   - 利用時間に応じた割引の自動計算

5. **問い合わせ機能**
   - オンラインフォームによる問い合わせ

6. **多言語対応**
   - 英語、中国語などの言語サポート

#### 3.2.2 管理者向け機能
1. **アクティブユーザーの確認**
   - 利用頻度の高いユーザーの識別

2. **予約の通知**
   - 新規予約、変更、キャンセルの通知

3. **月毎の利用状況の分析**
   - 予約状況のレポート生成
   - 収益分析

4. **問い合わせ・フィードバックの管理**
   - 顧客からの問い合わせ管理
   - フィードバック分析

## 4. 非機能要件

### 4.1 パフォーマンス
- ページ読み込み時間：2秒以内
- 同時アクセス数：最低50ユーザー

### 4.2 セキュリティ
- ユーザー認証
- データ暗号化
- 定期的なバックアップ

### 4.3 ユーザビリティ
- レスポンシブデザイン（スマートフォン、タブレット、PC対応）
- シンプルで直感的なUI
- アクセシビリティ対応

### 4.4 可用性
- 稼働率99.5%以上
- メンテナンス時間の最小化

### 4.5 拡張性
- 将来の機能追加を考慮した設計
- API設計による外部サービスとの連携可能性

## 5. 技術スタック

### 5.1 フロントエンド
- **フレームワーク**: Next.js
- **モバイル**: React Native

### 5.2 バックエンド
- **言語**: Golang
- **API**: RESTful API

### 5.3 データベース
- **DB**: PostgreSQL（検討中）

### 5.4 インフラ
- **クラウド**: AWS または GCP
- **CI/CD**: GitHub Actions（検討中）

## 6. 制約条件
- 予約は最低利用時間が2時間
- 11人以上での利用の場合は料金体系が異なる
- 仮予約は予約日の1週間前まで有効

## 7. 前提条件
- Google Calendarとの連携が必要
- 自動メール送信機能の実装

## 8. リスク
- 複雑な予約ルールの実装における課題
- 既存システムからの移行における顧客の混乱

## 9. 開発スケジュール
全6回のハンズオンセッション（各2時間×週1回）：
1. プロジェクトキックオフ＆ゴール設定
2. 画面構成と情報設計
3. ワイヤーフレーム作成
4. UI モックアップ設計
5. API 設計
6. DB 設計＆運用フロー検討

各セッションの成果物は Figma／OpenAPI 仕様書／ER 図などの形で GitHub リポジトリにまとめ、終了後にレビュー会を実施します。
